# 思维指南

> **用途**：拓展你的思考范围，捕捉那些你可能没想到的问题。

---

## 为什么需要思维指南？

**大多数缺陷和技术债，来源都不是“不会写”，而是“没想到”。**

- 没想到层与层之间会发生什么 -> 跨层 bug
- 没想到代码模式在重复出现 -> 到处都是重复代码
- 没想到边界场景 -> 运行时错误
- 没想到后续维护的人 -> 代码难以阅读

这些指南会帮助你**在编码前先问对问题**。

## 速查：何时该切换到思考模式

### 什么时候要考虑跨层问题

- [ ] 功能涉及 3 层及以上（API、Service、Component、Database）
- [ ] 数据格式会在不同层之间发生变化
- [ ] 同一份数据需要被多个消费者使用
- [ ] 你拿不准某段逻辑该放在哪一层

-> 阅读 [跨层思维指南](cross-layer-thinking-guide.md)

### 什么时候要考虑代码复用

- [ ] 你正在写的代码和已有实现很像
- [ ] 你看到同一模式重复了 3 次以上
- [ ] 你要在多个地方新增同一个字段
- [ ] **你正在修改任何常量或配置**
- [ ] **你要创建新的 utility/helper 函数** <- 先搜索！

-> 阅读 [代码复用思维指南](code-reuse-thinking-guide.md)

## 修改前规则（关键）

> **改任何值之前，先搜索一遍！**

```bash
# 搜索你准备修改的值
rg "value_to_change"
```

这个习惯本身，就能拦下大多数“忘了顺手更新 X”的 bug。

## 如何使用这个目录

1. **编码前**：快速浏览相关的思维指南
2. **编码中**：一旦感觉代码开始重复或变复杂，就回来翻指南
3. **出 bug 后**：把新的教训补充到对应指南里（别白踩坑）
