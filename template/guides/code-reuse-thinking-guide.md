# 代码复用思维指南

> **用途**：创建新代码前先停一下想想，这玩意是不是已经存在？

---

## 问题本质

**重复代码是一致性缺陷的头号来源。**

当你复制粘贴或重写已有逻辑时：
- bug 修复不会自动传播
- 行为会随着时间逐渐分叉
- 代码库会越来越难理解

---

## 写新代码之前

### 第一步：先搜索

```bash
# 搜索相似的函数名
rg "functionName"

# 搜索相似的逻辑
rg "keyword"
```

### 第二步：问自己这些问题

| 问题 | 如果答案是“是” |
|------|------------------|
| 是否已经存在类似函数？ | 直接复用或扩展它 |
| 这个模式是否在别处也出现？ | 遵循现有模式 |
| 这段能力是否应该成为共享工具？ | 把它放到正确的共享位置 |
| 我是不是正从别的文件复制代码？ | **停下**，抽成共享实现 |

---

## 常见的重复模式

### 模式 1：复制粘贴函数

**坏例子**：把一个校验函数复制到另一个文件

**好例子**：抽到共享工具里，需要的地方统一引用

### 模式 2：相似组件

**坏例子**：新建一个和已有组件 80% 相似的新组件

**好例子**：通过 props 或 variants 扩展现有组件

### 模式 3：重复常量

**坏例子**：在多个文件里定义同一个常量

**好例子**：只保留一个事实来源，其他地方统一导入

---

## 什么时候该抽象

**适合抽象的情况**：
- 相同代码出现 3 次以上
- 逻辑复杂到已经可能出 bug
- 多个人、多处功能都可能会用到它

**不适合抽象的情况**：
- 只用一次
- 简单到离谱的一行代码
- 抽象层本身比重复代码还复杂

---

## 批量修改之后

当你在多个文件里做了类似修改后：

1. **复查**：所有实例都改到了吗？
2. **搜索**：用 `rg` 再扫一遍，看看有没有漏网之鱼
3. **反思**：这块是不是该抽象了？

---

## 易踩坑：不同机制产出同一结果，但结构更新不同步

**问题**：当两套不同机制必须生成同一组文件时（例如：初始化阶段递归复制目录，而更新阶段手工 `files.set()`），目录结构的变更（重命名、移动、新增子目录）只会通过自动机制传播。手工那套会悄悄漂移。

**症状**：初始化一切正常，但更新时文件路径不对，或者有些文件直接漏掉。

**预防清单**：
- [ ] 迁移目录结构时，搜索所有引用旧结构的代码路径
- [ ] 如果一条路径是自动推导的（glob/copy），另一条是手工列举的，手工那条也必须同步更新
- [ ] 添加回归测试，比较两种机制的输出是否一致

- [ ] 没有本应共享却被复制粘贴的逻辑
- [ ] 常量只定义在一个地方
- [ ] 相似模式保持相同结构
