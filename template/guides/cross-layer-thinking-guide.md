# 跨层思维指南

> **用途**：实现之前先把数据如何跨层流动想明白。

---

## 问题本质

**大多数 bug 都出在层与层的边界，不是在层内部。**

常见的跨层 bug：
- API 返回的是格式 A，前端却按格式 B 处理
- Database 存的是 X，service 转成 Y 时把数据丢了
- 多个层用不同方式实现同一段逻辑

---

## 实现跨层功能之前

### 第一步：画出数据流

把数据如何流动列出来：

```
Source -> Transform -> Store -> Retrieve -> Transform -> Display
```

对每一段箭头，都问自己：
- 数据当前是什么格式？
- 哪些地方可能出错？
- 由谁负责校验？

### 第二步：识别边界

| 边界 | 常见问题 |
|------|----------|
| API ↔ Service | 类型不匹配、字段缺失 |
| Service ↔ Database | 格式转换、null 处理 |
| Backend ↔ Frontend | 序列化、日期格式 |
| Component ↔ Component | props 结构变化 |

### 第三步：定义契约

对每一个边界，都明确：
- 精确的输入格式是什么？
- 精确的输出格式是什么？
- 可能出现哪些错误？

---

## 常见的跨层错误

### 错误 1：默认假设格式一致

**坏例子**：不确认就默认日期格式一致

**好例子**：在边界处做显式格式转换

### 错误 2：校验散落各层

**坏例子**：同一件事在多个层重复校验

**好例子**：在入口统一校验一次

### 错误 3：抽象泄漏

**坏例子**：组件直接知道数据库 schema

**好例子**：每一层只了解自己的相邻层

---

## 跨层功能检查清单

实现前：
- [ ] 已画出完整数据流
- [ ] 已识别所有层边界
- [ ] 已定义每个边界的数据格式
- [ ] 已决定校验发生在哪一层

实现后：
- [ ] 已用边界场景测试（null、empty、invalid）
- [ ] 已验证每个边界的错误处理
- [ ] 已确认数据往返后不会丢失

---

## 什么时候要补充流程文档

以下情况建议补充详细的数据流文档：
- 功能跨越 3 层及以上
- 涉及多个团队
- 数据格式本身比较复杂
- 这个功能以前出过相关 bug
