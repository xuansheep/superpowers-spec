# 通用 Java 项目规范

---

## 概览

本目录提供一套适用于通用 Java 项目的工程规范。

阅读原则很简单：

- 先看 `index.md`，确认适用范围与阅读路径。
- 再按任务类型进入对应专题文档。
- 项目已有明确约定时，优先遵守项目真实约定；项目没有明确约定时，再回落到本规范。

---

## 阅读建议

按任务类型建议如下：

- 理解项目结构时，先看 `directory-structure.md`。
- 写普通业务代码时，先看 `code-style-guidelines.md`，再看 `engineering-guidelines.md`。
- 设计接口、服务边界或模块拆分时，重点看 `engineering-guidelines.md`。
- 改 Mapper、Repository、SQL 或表结构时，补看 `database-guidelines.md`。
- 涉及登录、鉴权、文件上传、外部回调、敏感数据时，必看 `security-guidelines.md`。

---

## 适用范围说明

本规范适用于以下场景：

- 通用 Java 后端项目
- 多模块或单模块 Java 工程
- 使用或不使用 Spring 的标准服务端应用
- 需要明确代码风格、工程边界、数据库和安全基线的团队

本规范不直接替代以下内容：

- 某个仓库已经稳定运行多年的历史约定
- 某个框架自己的强制规范
- 团队在特定业务或合规环境下额外加的专用规约

---

## 使用原则

- 优先遵守项目已有的明确约定，不要拿着“通用规范”去硬怼已经成熟的项目内规则。
- 当项目内部没有统一约定时，默认以本规范作为落地基线。
- 规则冲突时，先保证正确性与可维护性，再考虑书写偏好。
- 同一问题尽量只看一个最相关的专题，避免在多个文档里来回打转。

---

## 核心原则

这套规范背后的核心判断只有几条：

- 代码首先要可读，其次才是“写得像你自己”。
- 分层、事务、异常和日志边界要稳定，不要边写边发明新流派。
- 数据库和安全不是补丁章节，而是工程主线的一部分。
- 规范的目标是减少歧义和事故，不是增加表演型仪式感。
