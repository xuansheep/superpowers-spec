# 安全规范

- 面向通用 Java 项目的安全基线规则，聚焦输入校验、认证授权、敏感信息处理、错误暴露控制与常见风险规避。

---

## 概览

安全规范不是给审计看的门面文件，而是日常开发必须兑现的底线。

接口、文件、SQL、日志、配置、回调、鉴权这些环节，只要有一个地方脑子一热图省事，就可能把整个系统拖进坑里。安全问题不会因为“这是内部系统”就自动消失，这种侥幸心态本身就是漏洞温床。

---

## 输入校验与边界检查

- 服务端必须对所有外部输入做合法性校验，不信任客户端已经校验过的说法。
- 校验应覆盖必填性、长度、范围、格式、枚举取值和业务边界。
- 对文件名、路径、URL、排序字段、动态条件等高风险输入，优先使用白名单策略。
- 非法输入应尽早拒绝，不要让脏数据一路游到数据库或下游系统再炸。
- 数据在进入核心流程前应完成必要的规范化处理，避免同一含义出现多种脏格式。
- 对嵌套对象、集合和批量输入，也应做逐项校验，别只检查最外层壳子。

---

## 认证与授权

- 认证用于确认“你是谁”，授权用于确认“你能干什么”，两者不能混成一个模糊判断。
- 每个受保护操作都必须在服务端完成权限校验，不能相信前端传来的角色、用户 ID 或组织范围。
- 高风险操作应基于最小权限原则控制访问范围。
- 涉及跨租户、跨组织、跨用户数据访问时，必须显式校验数据归属关系。
- 权限判断应靠稳定规则和统一入口收敛，避免各业务点各写一套半截逻辑。
- 不要把“只有内部人会调这个接口”当成授权方案，那不是方案，那是做梦。

---

## 敏感信息处理

- 密码不得明文存储、明文传输或明文打印。
- 密钥、Token、验证码、证书、连接串等敏感配置不得硬编码在源码中。
- 敏感信息在日志、响应、导出和监控事件中应按最小暴露原则处理，能脱敏就脱敏。
- 手机号、邮箱、身份证号、银行卡号等个人信息应按业务必要性最小化存储和展示。
- 用于认证或签名的秘密材料应有明确轮换和失效策略。
- 测试数据和示例代码也不应夹带真实凭证，别把演示环境当藏污纳垢角落。

---

## 日志与错误暴露控制

- 对外错误信息应保持克制，不暴露堆栈、SQL 细节、内部路径、配置键值和框架内部对象结构。
- 对内日志应保留足够诊断信息，但仍需避免打印完整敏感数据。
- 异常链路应记录必要上下文，如请求标识、操作者、资源标识和失败动作。
- 登录、鉴权失败、权限拒绝、文件处理失败和外部回调异常等高风险事件应留下可审计记录。
- 统一异常处理机制应区分业务提示与系统故障，避免把内部实现细节原样吐给调用方。
- “为了排查先全量打印一下”这种操作很常见，也很蠢，尤其在生产环境里。

---

## 数据存储与传输安全

- 涉及敏感数据的传输链路应使用安全传输机制，避免明文暴露。
- 密码、令牌、签名密钥等应采用适当的加密、哈希或密钥管理方式处理。
- 安全随机值应使用可靠随机源生成，不要拿弱随机拼运气。
- 配置中心、环境变量、密钥文件等载体都应有访问控制，不能谁都能看。
- 数据落库前应评估是否需要脱敏、加密或分级存储。
- 外部系统交互时，应验证目标身份与协议安全性，不要只看“能连上”。

---

## 文件与外部资源安全

- 文件上传应校验大小、类型、后缀、内容特征和存储路径。
- 服务端生成文件路径时应避免路径穿越风险，不信任客户端提供的相对路径片段。
- 外部 URL、回调地址和重定向地址应校验来源与目标范围，避免 SSRF、开放跳转等风险。
- 对压缩包、脚本、模板和可执行内容应有更严格的处理策略。
- 下载、导出和预览接口应校验访问权限，不能只要知道地址就能拿。
- 与第三方系统交互时，应限制超时、重试和可访问目标范围，别把系统变成别人拿来打洞的代理机。

---

## 常见高风险模式

- SQL 注入：通过拼接 SQL 或动态字段直接把输入塞进查询。
- 命令注入：把外部输入拼进系统命令、脚本或模板执行链路。
- XSS：把未经处理的用户输入直接回显到页面或富文本内容中。
- 反序列化风险：反序列化不可信对象或信任外部传来的类型信息。
- 越权访问：只验证登录态，不验证资源归属和操作范围。
- 信息泄露：日志、错误页、导出文件或调试接口把内部信息裸出去。

---

## 常见错误

- 认为“前端已经校验过了”，于是后端直接放行输入。
- 把用户 ID、租户 ID、角色标识完全交给客户端传值决定。
- 出错时把完整异常堆栈、SQL、配置路径直接回给调用方。
- 在日志里打印密码、Token、身份证号或完整请求体，排查是排查了，合规也一起下葬了。
- 把密钥写进源码、配置文件仓库或示例脚本里，等于给未来的事故留纪念碑。
- 文件上传只校验后缀不校验内容，最后什么玩意儿都能进系统。
