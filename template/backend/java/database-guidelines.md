# 数据库规范

- 面向通用 Java 项目的数据库相关规则，聚焦命名、SQL、事务、索引、数据一致性与变更管理。

---

## 概览

数据库规范解决的是“数据是否可读、SQL 是否可维护、事务是否稳定、变更是否可控”。

很多项目代码表面看着整齐，数据库层却写得像野生摊位：命名混乱、`select *` 满天飞、事务边界飘忽、索引随缘。真出事故时，锅最后都得数据库背，但代码的人也别装无辜。

---

## 表与字段命名

- 表名、字段名、索引名统一使用小写 `snake_case`。
- 表名应体现业务语义，不使用无意义缩写和过度简写。
- 主键命名应保持稳定，一般使用 `id`；外键命名统一采用 `xxx_id` 风格。
- 布尔语义字段命名应清晰，避免 `flag1`、`status2` 这种糊弄命名。
- 字段名应表达真实业务含义，不要把程序内部临时概念直接扔进表结构。
- 避免使用数据库关键字、过长名称和容易混淆的缩写。

---

## SQL 编写原则

- 查询字段应显式列出，禁止滥用 `select *`。
- 别名应服务于可读性和映射稳定性，不要为了省事把语义全抹平。
- 查询条件、排序和分页逻辑应明确，避免依赖数据库默认行为碰运气。
- 涉及分页时，应保证排序条件稳定，否则翻页结果可能飘得像抽签。
- 复杂关联、聚合、统计 SQL 应保持结构清晰，必要时拆分查询或通过视图/中间层整理。
- 禁止通过字符串拼接直接组装 SQL 条件，参数必须走预编译或安全绑定机制。
- 能用批量操作解决的场景，不要循环单条执行把数据库当打桩机。

---

## ORM 与 DAO 使用边界

- 简单 CRUD 和明确条件查询可以交给 ORM 或数据访问框架处理。
- 复杂统计、聚合、多表报表型查询应使用显式 SQL，并保持可审查性。
- DAO 或 Repository 层只负责数据访问，不负责业务决策、事务编排和接口拼装。
- 不要在上层服务里散落 SQL 片段，也不要把业务规则塞进 Mapper 注释里伪装成“方便”。
- ORM 提供的是效率，不是替代思考；一旦 SQL 语义开始复杂，就该明确落回数据库表达。

---

## 事务边界

- 事务边界应放在服务层或清晰的业务协调层，不应放在控制器或底层 DAO 中随意开启。
- 只读操作应使用只读事务或明确的非事务语义，避免无意义加锁与资源占用。
- 事务范围应尽量小，只覆盖真正需要保持一致性的动作。
- 不要在长事务中混入远程调用、文件 IO 或不可控的外部等待操作。
- 回滚策略应明确，不能靠“出了错大概会回滚”这种玄学想象。
- 一致性要求高的流程，应优先清楚定义状态转换和失败补偿，而不是让事务无限变胖。

---

## 索引与性能

- 索引应围绕查询条件、排序条件、关联条件和唯一性约束设计。
- 不要为每个字段都建索引，索引是性能工具，不是装饰品。
- 高基数字段、常用过滤字段和稳定排序字段优先评估索引价值。
- SQL 上线前应关注执行计划，避免全表扫描、回表过多和低效 join。
- 应警惕 N+1 查询、循环查库和大结果集无节制加载。
- 批量写入、批量更新和归档任务应采用更合适的分批策略，避免单次事务过重。

---

## 空值、边界值与一致性

- `null` 的业务语义必须明确，不能同时表示“未知”“未填”“无效”“默认”。
- 能用默认值和约束表达的规则，不要完全依赖应用层口头约定。
- 唯一性、一致性和幂等要求应有数据库约束或稳定机制兜底，不能全靠代码自觉。
- 金额、计数、状态、时间等关键字段应明确精度、范围和默认行为。
- 状态变更和库存类操作应考虑并发冲突，不要把一致性赌给低流量时运气好。
- 删除、归档和历史保留策略应在模型层面明确，避免靠临时脚本救火。

---

## 变更脚本管理

- 表结构和关键数据变更应通过受控脚本管理，不要靠人工上线时现场发挥。
- 变更脚本应具备清晰命名、目的说明和执行顺序。
- 变更设计应优先考虑向后兼容，避免一步把旧逻辑和旧数据全打碎。
- DDL 与依赖它的应用代码应作为同一交付批次审查。
- 高风险数据修复脚本应先在受控环境验证，再进入生产执行流程。
- 对不可逆或影响范围大的变更，应提前设计回滚或补救方案。

---

## 常见错误

- `select *` 用习惯了，最后字段一多、索引一变，性能和映射一起炸。
- 把事务开在 Controller 或循环体里，写得像没见过生产流量。
- 远程调用塞进事务里，事务时间一长就开始拖垮连接和锁。
- 唯一性完全靠代码 if 判断，不做数据库约束，属于邀请并发问题进门。
- 线上大表无索引改造、无执行计划评估就直接跑 SQL，这不是勇，是莽。
- 数据修复不留脚本、不留说明，等于把系统历史交给口口相传。
